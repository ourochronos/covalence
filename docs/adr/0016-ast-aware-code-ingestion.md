# ADR-0016: AST-Aware Code Ingestion and Design Doc Bridge

**Status:** Accepted

**Date:** 2026-03-12

**Spec Reference:** spec/12-code-ingestion.md, spec/02-data-model.md

## Context

Covalence ingests its own source code, specs, and research papers. Currently, code files go through the same prose ingestion pipeline as natural language documents: Markdown normalization → fixed-size overlapping chunk windows → embedding → entity extraction. This creates three problems:

1. **Destructive chunking.** Splitting code at token boundaries bisects functions, structs, and impl blocks. A 512-token window that cuts a function in half produces two meaningless chunks that embed poorly and extract garbage entities.

2. **Semantic space mismatch.** Raw code syntax (`pub async fn embed_batch(&self, texts: &[&str]) -> Result<Vec<Vec<f32>>>`) produces vector embeddings in a completely different region of the latent space than the prose description of the same concept ("embeds a batch of text strings into vector representations"). Queries in natural language can't find code, and code can't find relevant spec sections.

3. **No structural linking.** There is no mechanism to connect spec concepts ("the ingestion pipeline should extract entities") to the code that implements them (`fn extract_entities()`), or to the research that describes the underlying algorithm ("HDBSCAN clustering"). The graph contains prose knowledge and code knowledge as disconnected islands.

These problems block the target emergent capabilities: research-to-execution verification, architecture erosion detection, whitespace roadmap, blast-radius simulation, and dialectical design critique.

## Decision

### AST-Aware Chunking via Tree-sitter

Replace token-count chunking for code sources with AST-boundary chunking using Tree-sitter. Code is split at logical boundaries: functions, structs, traits, impl blocks, modules. Tree-sitter provides:

- Language-agnostic API with per-language grammars
- S-expression queries for extracting specific AST node types
- Incremental parsing for re-ingestion efficiency
- Structural edges (call graph, type references, module hierarchy) for free

Start with Rust and Go grammars (the languages Covalence is written in). The design is language-extensible.

### Semantic Summary Wrapper

For each AST chunk, generate a natural language summary of the code's business logic by passing it to a fast LLM (Gemini Flash or similar). Embed the summary, not the raw code. This places code entities in the same vector space as prose concepts.

The summary is stored as `semantic_summary` in the code Node's JSONB `properties`. The Node's `embedding` is computed from this summary. An `ast_hash` (SHA-256 of AST structure, ignoring whitespace/comments) enables incremental updates: only re-summarize when the code's structure actually changes.

### Component Bridge Layer

Introduce a `Component` entity type that bridges spec/design doc topics to code entities. Components are logical groupings (e.g., "Ingestion Pipeline", "Search Fusion", "Entity Resolution") connected via typed edges:

- `IMPLEMENTS_INTENT`: Component → Spec Topic Node (upward, toward design intent)
- `PART_OF_COMPONENT`: Code Node → Component (upward, toward logical grouping)
- `THEORETICAL_BASIS`: Component → Research Node (lateral, toward academic foundation)

Components can be created manually (bootstrapped from known architectural boundaries), auto-detected from design doc topic clusters, or inferred from module path patterns.

### Code-Specific Node Types and Edge Types

Code entities are represented as Nodes with specialized `node_type` values: `code_function`, `code_struct`, `code_trait`, `code_module`, `code_impl`, `code_type`, `code_test`.

Structural edges extracted from AST: `CALLS`, `USES_TYPE`, `IMPLEMENTS`, `CONTAINS`, `DEPENDS_ON`.

Analysis edges generated by cross-domain algorithms: `SEMANTIC_DRIFT`, `COVERAGE_GAP`.

## Consequences

### Positive

- **Cross-domain search works.** "How does entity resolution work?" returns spec statements, research paper claims, AND code function summaries — because they share a vector space.
- **Architecture erosion is measurable.** Cosine distance between Component descriptions and code semantic summaries quantifies drift.
- **Coverage gaps are visible.** Graph traversal reveals orphan code (no spec link) and unimplemented specs (no code link).
- **Blast radius is computable.** Semantic chains from code to spec to research enable impact analysis that follows meaning, not just imports.
- **Incremental updates are efficient.** AST hash comparison means re-ingestion only re-summarizes structurally changed functions.
- **The meta-loop improves.** Covalence reasoning about its own code through its own graph closes the self-improvement loop.

### Negative

- **Tree-sitter dependency.** Adds a native dependency for each supported language grammar. Build complexity increases.
- **LLM cost at ingestion.** Every function/struct needs a semantic summary pass. For Covalence's ~600 public items, this is ~600 LLM calls (cheap per-call, but nonzero).
- **Component linking requires bootstrapping.** Initial Component creation and edge wiring requires manual effort or heuristic-based auto-detection.
- **Summary quality is a new failure mode.** If the LLM generates a poor summary, the code entity will be poorly embedded and poorly searchable.

## Alternatives Considered

### 1. Embed raw code directly
Embedding raw syntax produces vectors in code-specific latent regions that don't overlap with prose. Cross-domain queries fail silently — the vectors are in different neighborhoods. Rejected because it defeats the core purpose: bridging code and spec.

### 2. Fine-tuned code embedding model (CodeBERT, CodeSage)
Specialized models embed code into a code-aware vector space. But the goal isn't code-to-code similarity — it's code-to-prose similarity. A fine-tuned code model would still place code embeddings far from natural language spec embeddings. Rejected because it doesn't bridge the semantic gap.

### 3. Regex-based function extraction (no Tree-sitter)
Could extract functions via regex (`fn\s+\w+`) without the Tree-sitter dependency. But regex can't handle nested braces, lifetime annotations, macro invocations, or module hierarchy. It would miss impl blocks entirely and produce broken chunks on complex code. Rejected for insufficient accuracy.

### 4. Skip code ingestion entirely
Just ingest specs and research, not code. This eliminates the bridging problem but also eliminates the target capabilities: no erosion detection, no coverage analysis, no blast radius. Rejected because these capabilities are the whole point.
